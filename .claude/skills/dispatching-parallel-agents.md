# SKILL: Dispatching Parallel Agents (병렬 에이전트)

**트리거:** 독립적인 작업 2개 이상 동시 존재 시

## 언제 사용하는가

- BE와 FE가 서로 다른 파일을 수정하고 독립적으로 진행 가능할 때
- 3개+ 테스트 파일이 각각 다른 원인으로 실패할 때
- 각 문제가 서로 context 없이 해결 가능할 때

## 언제 사용하지 않는가

- 실패들이 관련되어 있음 (하나 고치면 다른 것도 해결 가능)
- 에이전트들이 같은 파일을 수정해야 함 → **직렬 순서 필수** (CONTEXT.md 비자명적 결정 참조)
- 공유 상태가 있음 (DB 스키마 등)

## 프로젝트 에이전트 타입

| 에이전트 | 역할 |
|---------|------|
| `be-feature-builder` | BE 구현 (Kotlin/Spring Boot) |
| `fe-feature-builder` | FE 구현 (React 19 + TypeScript) |
| `qa-reviewer` | 코드 리뷰 + BE↔FE 계약 정합성 |
| `design-reviewer` | 신규 UI 컴포넌트 스펙 설계 |

## 병렬 에이전트 프롬프트 구조

```markdown
## 브랜치
이미 생성됨: <브랜치명>
git fetch origin && git checkout <브랜치명>

## 담당 범위
[구체적 파일/서브시스템]

## 작업
[명확한 구현 요청]

## 제약
[수정하면 안 되는 코드 범위]

## 반환 필수
1. 구현 파일 목록
2. [BE면 API 스펙 / FE면 컴포넌트 인터페이스]
3. 커밋 해시
```

## 병렬 실행 시 주의

- 같은 브랜치에서 병렬 실행 시 push 충돌 위험 → 순서 조율 필요
- BE 완료 후 API 스펙을 FE에 전달하는 경우 → 병렬 불가, 순차 실행
