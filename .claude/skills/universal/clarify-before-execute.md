# SKILL: Clarify Before Execute (실행 전 명확화)

**트리거:** 요청의 의도, 범위, 완료 기준이 한 문장으로 확정되지 않은 모든 상황

## 절대 규칙

```
의도가 한 문장으로 확정되기 전까지 파일을 열지도, 코드를 검색하지도 않는다
```

"아마 이런 뜻이겠지"는 가정이다. 가정 위에 구현하면 재작업이 생긴다.

## 라우팅 원칙 (질문 전 먼저 판단)

모든 불명확한 항목을 두 가지로 분류한다:

| 분류 | 처리 방법 |
|------|----------|
| **코드에서 답할 수 있는 것** | 코드 읽고 직접 해소 — 사용자에게 묻지 않는다 |
| **인간 판단이 필요한 것** | 질문한다 |

예시:
- "현재 어떤 패턴을 쓰고 있나요?" → 코드 읽고 해소
- "이 기능의 범위를 어디까지 볼까요?" → 사용자에게 질문

## 4단계 프로세스

### Phase 1: 코드에서 해소 가능한 항목 먼저 처리

요청과 관련된 파일, CONTEXT.md, 최근 커밋을 읽고 코드에서 확인할 수 있는 사항을 먼저 해소한다.
코드 탐색 후에도 불명확한 항목만 Ambiguity Ledger에 기록한다.

### Phase 2: Ambiguity Ledger 구성

불명확한 항목을 우선순위 순으로 목록화한다.

불명확 축 우선순위 (위에서 하나만 고른다):
1. **목표** — 무엇을 달성하려는가
2. **범위** — 어디까지 포함/제외인가
3. **제약** — 건드리면 안 되는 것이 있는가
4. **완료 기준** — 언제 "됐다"고 판단하는가
5. **기존 맥락** — 연관된 기존 기능/PR이 있는가

### Phase 3: 질문 (한 번에 하나씩)

가장 중요한 불명확 축 하나만 질문한다.
같은 축 안에 질문이 여러 개면 **답에 따라 아키텍처·데이터 모델이 바뀌는 질문부터** 묻는다
(굳으면 비싼 결정 우선).

형식:
```
현재 이해: {요청을 한 문장으로}
막힌 결정: {가장 중요한 불확실성}
추천 답안: {있으면 제시, 없으면 생략}
질문: {한 가지만}
```

답변을 받으면:
- Ledger에서 해당 항목을 제거
- 남은 항목 중 다음 우선순위 하나로 반복
- 모두 해소되면 Phase 4로

### Phase 4: Restate Gate (실행 전 필수)

모든 항목이 해소되면 실행 전에 반드시 재진술 + 승인을 받는다.

```
## 실행 계획 확인

목표: {한 문장}
범위: {포함 / 제외}
완료 기준: {측정 가능한 조건}
제약: {있으면}

### 결정 테이블 (질문이 1개라도 있었으면)
| 결정 | 선택 | 근거 |
|------|------|------|
| {결정 항목} | {선택한 답} | {사용자 답변 / 코드 확인} |

이 이해가 맞으면 진행합니다. 수정할 부분 있나요?
```

결정 테이블은 이후 구현(또는 builder 프롬프트)의 기준점으로 그대로 전달한다.

사용자가 명시적으로 승인(OK / 맞아 / 진행해 등)하기 전까지 실행하지 않는다.

## 빨간 깃발

- "아마 이런 뜻이겠지" → Phase 2로 돌아가라
- 여러 질문을 한 번에 → 한 가지씩으로 쪼개라
- Restate 없이 바로 구현 → Phase 4 먼저
- 승인 없이 "진행할게요" → 금지

## 종료 조건

Restate Gate에서 사용자 승인 완료 → 스킬 종료, 구현 시작
