# SKILL: Systematic Debugging (체계적 디버깅)

**트리거:** 버그, 테스트 실패, 예상치 못한 동작 — 모두

## 철의 법칙

```
근본 원인 조사 없이는 수정 없다
```

"빠른 수정" 생각이 드는 순간이 가장 위험하다. 특히 긴급 상황일수록 체계적 접근이 더 빠르다.

## 4단계 프로세스

**Phase 1: 근본 원인 조사 (수정 전 필수)**
1. 에러 메시지 완전히 읽기 (스택 트레이스 포함)
2. 재현 가능하게 만들기
3. 최근 변경사항 확인 (`git diff`, 최근 커밋)
4. 여러 컴포넌트가 있다면 → 각 경계에 진단 로그 추가 후 실행하여 어디서 깨지는지 확인

**Phase 2: 패턴 분석**
- 작동하는 유사한 코드 찾기
- 레퍼런스 구현과 비교 (완전히 읽기, 스킵 금지)
- 차이점 목록화

**Phase 3: 가설 및 테스트**
- 단일 가설 수립: "X가 근본 원인이다. 왜냐하면 Y"
- 가장 작은 변경으로 테스트
- 한 번에 하나씩만

**Phase 4: 구현**
- 실패하는 테스트 케이스 먼저 작성 (TDD 적용)
- 단일 수정
- 검증

## 3번 이상 수정 실패 시

수정을 계속 시도하지 말고 **아키텍처 자체를 재검토**하라.
각 수정이 다른 곳의 새 문제를 드러내고 있다면 → 근본적으로 잘못된 구조다.
막히면 즉시 사용자에게 보고하고 방향을 묻는다.

## 빨간 깃발

- "일단 이거 바꿔보고..." → Phase 1로 돌아가라
- 원인 이해 없이 여러 수정을 동시에 적용 → 금지
- "아마 X 때문인 것 같다" → 가설을 증거로 검증하라

## 프로젝트 특유 패턴

- 생성자 변경 시 → `@Mock` 누락 여부 즉시 확인
- `verify()` 실패 시 → 파라미터 타입 불일치 확인 (Kotlin null 포함)
- Flyway 오류 시 → V* 버전 번호 중복, `repair()` 선행 여부 확인
