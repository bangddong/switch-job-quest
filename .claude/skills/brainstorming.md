# SKILL: Brainstorming (설계 먼저)

**트리거:** 새 기능, 컴포넌트 추가, 동작 변경 **모두**

## 절대 규칙

코드 한 줄도 쓰기 전에 설계를 완성하고 사용자 승인을 받는다.
"이건 너무 간단해서 설계가 필요 없다"는 생각이 드는 순간이 가장 위험하다.

## 체크리스트 (순서대로)

- [ ] 프로젝트 현재 상태 파악 (파일, `.claude/CONTEXT.md`, 최근 커밋)
- [ ] 명확화 질문 — **한 번에 하나씩만** (멀티초이스 우선)
- [ ] 2~3가지 접근법 제안 (트레이드오프 포함, 추천 명시)
- [ ] 설계 문서 작성 후 사용자 승인
- [ ] `.claude/plans/YYYY-MM-DD-<topic>-design.md` 저장
- [ ] 사용자 검토 후 → `writing-plans` 스킬로 전환

## 설계 원칙

- **YAGNI 철저히 적용** — 필요 없는 기능은 설계에서 제거
- **각 컴포넌트는 하나의 책임만** — 명확한 인터페이스, 독립적 테스트 가능
- 기존 코드베이스 패턴 따름 (Port & Adapter, props drilling 등)

## 리스크 선제 제시 형식

구조 변경 / 멀티스텝 계획 시 반드시 포함:

```
> 리스크: <이 접근의 가장 큰 약점>
> 전제 조건: <이 제안이 맞으려면 성립해야 하는 가정>
> 대안: <다른 접근이 있다면>
```
